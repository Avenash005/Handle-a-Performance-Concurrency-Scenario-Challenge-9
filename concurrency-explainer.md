# Concurrency Explainer&#10;&#10;**Your name:** [Your Name]&#10;**Date:** [Current Date]&#10;&#10;---&#10;&#10;## The Root Cause — Why Check-Then-Insert Fails&#10;&#10;The root cause is the 'check-then-insert' anti-pattern. In the original broken code, the booking endpoint first queries the DB with `prisma.booking.findFirst({where: {seatId, showId}})` to check if the seat is taken. If no existing booking, it then `prisma.booking.create({data: {userId, seatId, showId}})` .&#10;&#10;The problem is these are two separate DB operations with a timing gap. Under concurrent load (e.g., two requests at midnight flash sale), both requests can: 1) Run findFirst simultaneously and both see no booking (seat available). 2) Milliseconds later, both reach create and both succeed, double-booking the seat. This is a race condition – the outcome depends on timing, non-deterministic. App-level checks can't prevent it because the gap exists between operations; no mutex at app level.&#10;&#10>## Why the Unique Constraint Fixes It&#10;&#10;The @@unique([seatId, showId]) constraint moves uniqueness enforcement to the database schema level. DB constraints are atomic – create operations are executed as single transactions. The first concurrent create succeeds. The second hits the unique index violation (P2002), rejected immediately. No app code gap. App checks can't be atomic across requests, but DB handles it at storage layer.&#10;&#10;## Why Rate Limiting Alone Is Not Enough&#10;&#10;Rate limit (10/min/IP) stops floods/bots exhausting resources, returning 429 before DB hit. But for legitimate concurrent users (different IPs), e.g., two fans one req each within limit, both pass rate limit, race through, both create succeeds without constraint. Rate limit defends volume, not races between valid requests.&#10;&#10;## What P2002 Means and Why 409&#10;&#10;P2002 is Prisma's code for unique constraint violation. Expected when race loser hits @@unique. 409 Conflict correct: request valid but conflicted with existing resource (seat taken). Not 400 (valid input), not 500 (not error, expected race outcome). Clean UX: "Seat taken, try another".&#10;&#10;**Total word count:** 380
